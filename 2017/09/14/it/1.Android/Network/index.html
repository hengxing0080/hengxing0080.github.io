Android —— 网络编程总结

发表于 2017-09-14 | 分类于 IT - Android |


在学习各种开源网络框架如okhttp还是retrofit，进阶之前还是先回顾总结一下Android原生提供的网络请求。
温故而知新，基础要打好，也要总结好！

访问网络

网络请求的几种方式

HttpURLConnection

来源：是java语言自带的API

特色：
1.手工的面向过程的HTTP协议编程
2.采用Url和HttpURLConnection提交数据到服务器

缺点：
1.需要注意HTTP访问的底层细节问题，比较麻烦
2.需要考虑子线程访问和主线程更新的问题
主线程中不能进行耗时的操作–访问网络ANR，像这种耗时的操作必须放置到子线程内部执行，因为子线程不能更新UI,只有主线程能够更新UI

历史：
在Android API Level 9（Android 2.2）之前之能使用DefaultHttpClient类发送http请求。
DefaultHttpClient是Apache用于发送http请求的客户端，其提供了强大的API支持，而且基本没有什么bug，但是由于其太过复杂，Android团队在保持向后兼容的情况下，很难对DefaultHttpClient进行增强。
为此，Android团队从Android API Level 9开始自己实现了一个发送http请求的客户端类——–HttpURLConnection。

相比于DefaultHttpClient，HttpURLConnection比较轻量级，虽然功能没有DefaultHttpClient那么强大，但是能够满足大部分的需求，所以Android推荐使用HttpURLConnection代替DefaultHttpClient，并不强制使用HttpURLConnection。

但从Android API Level 23（Android 6.0）开始，不能再在Android中使用DefaultHttpClient，强制使用HttpURLConnection。

GET和POST的区别

GET提交数据到服务器

原理：组拼url的路径在路径后面拼装 name=?&password=?

缺点：1. 不安全 2.数据大小长度有限制不能超过4K
优点：代码写起来很简单

POST提交数据到服务器

原理：数据是以流的方式写给服务器

优点：1.安全（用工具其实也能拦截） 2. 数据大小长度没有限制
缺点：代码编写麻烦

代码的不同之处

提交数据到服务器和获取服务器中返回数据的中文乱码的解决

网络请求发送的参数必须进行URL编码
乱码问题,保证编解码一致就不会乱码

web容器默认采用的编码是iso-8859-1
还有不默认的情况, 如果当前的内容在iso里找不到,就会当前运行的本地码表去找这些内容

android系统默认的码表utf-8
公司开发也都是utf-8编码，因为要用很多第三方的（也是U8编码），所以方便
为了保证应用程序不显示乱码, 保证android端和服务器端采用相同的文件编码集.

关于参数中有中文
URL编码（解决乱码）：不管包不包含中文，请求参数都要进行URL编码的操作，因为请求只能传输ISO8859-1，中文必须URL编码

HttpClient

作者：Apache的一个开源项目（Apache 开源项目组织）

简介：
相当于一个轻量级的纯java代码浏览器
面向对象思想的封装来调用方法, 逻辑已经搞定了, 比之前简单多了
之前的写法(post)写的很复杂. 必须对浏览器非常熟悉,知道如何组拼字符串,太麻烦了
所以采用httpclient编写的post代码很方便, 面向对象封装非常好, 不需要知道HTTP细节
安卓, Java Web , JavaSE 开发都可以使用httpclient

优点：
HttpClient 改变了URLConnetion编程面向过程的方式变为了面向对象的方式访问网络,隐藏了底层HTTP协议实现的细节,方便代码的开发

现状：
API 23（6.0）废弃
但仍然还是有方法使用
Android API 23中废弃了HttpClient的解决办法

步骤：
1.打开浏览器
HttpClient client = new DefaultHttpClient(); //别直接newHttpClient，要复写一堆方法太复杂，所以new实现类
2.输入数据
HttpGet 只需要设置请求的路径就ok
HttpPost 不仅需要设置请求的路径,还需要设置数据的内容
httpPost.setEntity(new UrlEncodedFormEntity(parameters,”utf-8”));
3.敲回车
client.execute(get/post)
4.得到请求的相应
response.getStatusLine().getStatusCode(); 得到状态码
response.getEntity().getContent(); 得到数据实体

AsyncHttpClient

来源：android的开源项目(在github下载)

优点：
1.隐藏了子线程访问主线程更新的问题
2.使用简单，我们只需要告诉他,访问什么地址，带着哪些参数，访问完成后回调什么方法

简介：
对appache的HttpClient进行了进一步封装
面向对象的方式去访问网络,并且隐藏了细节,
不需要自己开辟子线程,不需要考虑消息机制,
只要告诉程序 要访问的地址,想要带过去的参数,成功和失败时的操作,剩下的事情由程序的底层去实现

异步帮我们做好了，get方法底层开辟了子线程去访问path地址，
访问完用消息机制把消息传给handler，由handler调用写好的onSucess和onFailure方法
就是底层把子线程给隐藏掉了
它巧妙了回避掉了子线程的问题，不用考虑它，直接写代码就行了

这个开源项目使用非常广泛, 基本安卓公司做互联网应用采用的HTTP请求框架都是基于这个框架或者做的改造
所以写法版本一定要掌握
写网络请求通常要用到异步（新开个线程去操作），因为访问网络是个耗时的操作
最简单的方式，推荐使用

HttpUtils

属于开源框架XUtils
OnSucess和onFailure方法都是在主线程里运行

网络异常

异常名称：android.os.NetWorkOnMainThreadException

网络在主线程异常
安卓在设计时就定义了不允许在主线程里面进行网络请求
这是新版本的异常
安卓版本 2.3.7 运行没有这个异常
不过在安卓版本4.1的模拟器就出现了这个异常
访问网络是一个可能比较耗时的操作.
在低版本的android系统谷歌没有考虑到用户体验的问题.
在android4.0版本开始,谷歌android系统开始检查,如果在主线程里面访问网络,强制的抛出异常.

在android应用程序里面所有的用户交互的操作都是在主线程里面执行
按钮点击事件,界面的渲染和显示,触摸事件
就是点击事件属于主线程

主线程做的事：
响应事件
UI显示和更新

文件下载

多线程并发下载的原理

为什么多线程并发下载可以提高速度？
服务器的资源有限, 客户端线程越多单位时间内, 抢到的服务器的资源就越多, 下载的速度就越快

下载的速度是由本地带宽决定：1Mbps 128KB/s
下载的带宽还由服务器的带宽决定：迅雷, p2p

并不是开的线程越多下载速度越快，安卓下3~5线程(开销最小,效率越高)

推荐使用exe文件作为下载测试的文件

多线程文件下载

原理

获取要下载的文件的大小
在本地创建和要下载的文件大小相同的文件
将要下载的文件分割成多个部分,开启多个线程分别进行下载,将下载的数据都放置到之前创建的空文件中
最终所有的线程都下载完成时,整个文件也就下载完成了

多线程断点下载的原理

记录上一次你下载了多少的数据,再去开启从上一次的位置接着下载

为每一个多线程下载文件的线程都创建一个文件,实时保存下载到的位置
如果程序突然停止了,下次再启动时,去读取文件,获知当前线程应该从哪里接着下载,就可以实现断点下载功能了
