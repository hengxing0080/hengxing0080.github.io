Android —— 关于四大组件的总结

发表于 2017-09-18 | 分类于 IT - Android |


一个apk就是一个应用程序
清单文件里面包含的所有的组件(Activity,service,content provider, broadcast receiver)
一个应用程序是由一堆组件组合在一起的,这些组件需要在清单文件下配置

这四大组件分别是：
1.Activity 活动界面(与用户交互的UI界面 ,在里面显示一些控件和响应一些点击事件)
2.Service 服务(后台长期运行的组件, 理解成没有界面的Activity)
3.BroadcastReceiver 广播接受者(用来接收或者处理操作系统发来的广播消息,或者是自己的应用程序发来的自定义广播 )
4.ContentProvider 内容提供者(把自己应用程序私有的数据库, 暴露给别的应用程序让别的应用程序可以对自己私有的数据库实现增删改查)

它们的共性：
四大组件都需要上下文（四大组件new对象没有意义，因为没有上下文）
四大组件都需要在清单文件下配置
四大组件都在主线程运行

注意：
不要用四大组件去实现接口 一是组件的对象都比较大，实现接口比较浪费，而且让代码更不易读和理解；
另外更重要的是导致多方引用，可能会引发内存泄露。

Activity

名称：活动

一般的,一个用户交互界面对应一个activity

activity 是Context的子类

实现了layoutInflater.Factory2
LayoutInflater是个打气筒，把xml转变成view对象挂载在我们的窗体上

同时实现了window.callback和keyevent.callback这2个接口(窗体的回调，可以处理与窗体用户交互的事件.）
像Service和广播接受者这些组件都没有实现这些接口，所以不能和用户交互

如项目开发常用的Activity的有：
1.BaseActivity（如果界面有共同的特点或者功能的时候,就定义一个它，如界面的初始化操作，设置进度条的显示与销毁，触摸事件）
2.FragmentActivity
3.ListActivity（获取ListView）
4.TabActivity（获取TabHost）
5.PreferenceActivity（设置相关的Activity)

意图（页面跳转和数据传递）

API：Intent

功能：
想让安卓系统做什么事情,就定义出意图
意图就是2个界面的桥梁

设计原则：
Android中鼓励在程序设计中尽量减少组件的耦合，因此Android中提供了一系列减少耦合的做法，表层我们熟悉的比如xml的布局文件和Activity分离，除此之外，还有一个重要的内容就是intent。
它就好比是各个组件中互相通讯的纽带，在四大组件中除了内容提供之外，其他的三大组件都可以通过intent来通讯。
除此之外，intent还有启动其他组件的作用。
比如启动Activity，服务，绑定服务，通过intent还可以在广播之间传递数据。

意图的2种跳转方式

Intent可以划分为：
显式意图和隐式意图
显式意图通常主要是启动本应用中的Activity之间的数据
而隐式意图则常见于启动系统中的某些特定的动作，比如打电话，或者是跨应用的Activity启动。

显示意图

简介：
调用Intent.setComponent()或Intent.setClass()方法明确指定了组件名的Intent为显式意图，
显式意图明确指定了Intent应该传递给哪个组件。
显式的指定要开启的界面的包名和类名 ( 这种方式免去声明清单文件的苦恼 )
就说直接创建意图, 指定访问哪个应用哪个页面,发送意图
不指定动作，不指定数据，直接指定要激活的组件

优点：
需要明确的指定要打开哪个页面
代码编写很简单，无需额外的配置。只需要指定要激活组件的字节码就可以了。
不需要在清单文件中配置 意图过滤器
安全性很高,性能要高

缺点：
代码的耦合性比较强，最好是 高内敛 低耦合
不可以跨着应用调用页面（如果不希望其他应用能够访问又需要在当前应用中访问,就不配置意图过滤器,用显示意图的方式调用即可）

应用场景：
如果你要激活自己应用程序的里面的界面（自己应用程序开启自己的界面 ）
推荐使用显式意图,因为效率高, 直接找到字节码就创建出来了，效率很高

作用域：src

写法：

隐式意图

简介：
没有明确指定组件名的Intent为隐式意图

不需要明确的指定打开哪个页面,只要被意图过滤器拦截到就可以响应这次意图.
通过指定应用程序的动作行为,数据和类别来去开启界面

Android系统会根据隐式意图中设置的动作(action)、类别(category)、数据（URI和数据类型）找到最合适的组件来处理这个意图。

原理：
在代码里面设置intent的action和data和Cateory，框架通过解析清单文件找到Activity然后开启

优点：
可以跨着应用调用页面.非常的方便

缺点：
安全性不高,只有希望被其他应用可以调用的页面才会配置意图过滤器.
在访问时需要找清单文件进行比对,性能比稍低

使用注意：
1.清单文件里面这个界面必须配置有意图过滤器(intent-filter)
根据intent-filter开启界面
开启更加灵活,精准,并且可以携带匹配的数据
2.如果一个意图被多个意图过滤器所拦截到，会让你选择一个应用

应用场景：
开启别人应用程序的界面
自己的某个界面，想让别的应用程序开启

应用步骤：
a页面 调用 b页面,
在b页面的配置中配置意图过滤器
a程序中发送匹配的意图
b页面拦截到意图后启动

作用域：
在清单文件注册

写法：

跳转时遇到的权限问题：
在外部跳转（尤其是第三方应用跳转系统应用）时，相信大家肯定遇到了个权限问题

这个是因为2个应用不在同一个用户组里，是跨进程跳转。
需要在AndroidManifest.xml中要跳转的activity中加入一个属性android:exported=”true”，允许外部跳转，此时能够跳转成功。
但是，这个属性的介绍是这样的：

http://developer.android.com/guide/topics/manifest/activity-element.html#exported

当这个值为“false”时，只能在同一个应用或者有相同的user id（比如跳转系统应用，则要添加system uid,并签名）的应用里被启动。
当为“true”，不管什么组件，都能启动这个activity。

这个属性的默认值依赖于这个activity是否声明了intent-filter,如果没有，则为false；否则为默认为true.

意图传递数据

在页面跳转时传输数据

意图传递数据
从一个界面调用到另外一个界面就用意图
多个页面跳转就用意图来实现 (应用内A页面打开B页面)

写法：
方式1:利用意图自带的data方式传输数据,此种方式适合传输简单的字符串

方式2:可以通过Intent的putXXX方法传递数据到各大组件中，在通过getXXX取出
支持的数据类型：
八种基本数据类型
字符串和一维数据
所有实现过Serializable接口的类的对象 （序列化到硬盘文件,android自己实现的序列化的类）
所有实现过Parcelable接口的类的对象（序列化到操作系统的公共内存里）

关于Serializable和Parcelable：
常见用来传递list或者对象
Serializable的方式较为简单，但由于会把整个对象进行序列化，因此效率方面会比Parcelable方式低一些，用于数据持久化

Parcelable在内存中进行，通过JNI
整个读写全是在内存中进行，主要是通过malloc()、realloc()、memcpy()等内存操作进行，所以效率比JAVA序列化中使用外部存储器会高很多

所以在通常情况下还是更加推荐使用Parcelable的方式来实现Intent传递对象的功能。

结果码 : resultCode
它们是int类型参数，很方便，能让我们区别出来数据来源于谁，或者是从谁那里返回来的
应用程序里面需要开启2个界面, 要获取它的返回值, 这时就需要获取请求码了

两个Activity之间跳转时必然会执行的是哪几个方法?
一般情况比如说有两个activity,分别叫A,B
当在A里面激活B组件的时候,A 会调用 onPause()方法,
然后B 调用onCreate() ,onStart(), OnResume() ,
这个时候B覆盖了窗体, A会调用onStop()方法.
如果B是个透明的,或者是对话框的样式, A就不会调用onStop()方法
如果忘了就告诉面试官看生命周期的哪个方法先执行的解决方案：在各个方法打个log

场景分类：
1.完整生命周期 (entrie lifetime)
oncreate –> onstart –> onresume –> onpause –> onstop –> ondestroy

2.可视生命周期 (visible lifetime) (可以看到和不可以看到的过程)
onstart –> onresume –> onpause –> onstop

3.前台生命周期 (foreground lifetime)
onresume –> onpause // 界面用户仍然可见，但是失去焦点

4.横竖屏切换的生命周期
用途:游戏开发中屏幕的朝向都是写死的，如97拳皇
默认横竖屏切换Activity被销毁又重新创建
这是因为,很多应用在横竖屏的情况下,布局是完全不同的,所以在切换时需要重新绘制Activity
可以把屏幕的朝向写死,这样Activity就不会重新销毁在创建
Manifest清单文件 / activity节点下

或者让操作系统忽略掉屏幕切换的变化

注意：
要考虑到Activity和进程被杀掉的情况 如了通常情况退出Activity外，还有Activity因其他原因被杀的情况，比如系统内存过低，系统配置变更，有异常等等，要考虑和测试这种情况，特别是Activity处理重要的数据时，做好的数据的保存。

Activity的四种状态：
1.激活状态
一个新的Activity启动入栈后，它在屏幕最前端，处于栈的最顶端，可以和用户交互。

2.暂停状态
当Activity被另外一个透明或者Dialog样式的Activity覆盖时的状态。
此时它依然与窗口管理保持连接。系统继续维护其内部状态，所以它仍然可见，但它已经失去了焦点故不可与用户交互。

3.停止状态
当Activity被另外一个Activity覆盖、失去焦点并不可见

4.销毁状态
Activity被系统杀死回收或者没有被启动

任务栈

相关文章：
深入讲解Android中Activity launchMode
基础总结篇之二：Activity的四种launchMode

又名：后退栈 / Activity栈
英文名：Task stack / Back stack

简介：
记录用户操作的行为，将打开的Activity按照一定的方式保存到栈中,让用户可以一层层的退出Activity,提高了用户的体验
按Activity打开的顺序就存进了栈里, 退出时一层一层往外退出

在Adroid应用中，可以有多个Activity，这些Activity组成了Activity栈，当前活动的Activity位于栈顶，之前的Activity被压入下面，成为非活动Activity，等待是否可能被恢复为活动状态。

为了记录用户开启了那些activity，记录了这些activity开启的先后顺序，google引入任务栈概念，帮助维护好的用户体验。

目的 : 使用户具有更好的用户体验

特点：
1.一个应用程序一般都是由多个activity组成的。
2.任务栈记录存放用户开启该的activity的。打开Activity就进栈，退出Activity就出栈
3.一个应用程序一被开启系统就给他分配一个任务栈，当所有的activity都退出的时候，任务栈就清空了。
4.每个任务栈都有一个编号，任务栈的id是一个integer的数据类型自增长的。
5.在android操作系统里面会存在多个任务栈，一个应用程序一个任务栈，,这个应用中的所有Activity默认公用这个任务栈
6.桌面应用和一般的应用程序是一样的，任务栈的行为也是一样。
7.默认情况下，关闭掉一个应用程序，清空了这个应用程序的任务栈。应用程序的进程还会保留。

任务: 用户操作的界面.
任务栈:记录用户操作的先后顺序,记录用户操作软件的行为,如果任务栈清空,应用程序就被关闭了.

Activity的任务栈有4种启动模式 :

standard 标准模式

默认的启动模式

开启这个Activity,系统就会new一个新的Activity,
这个Activity的引用就会出现在任务栈的顶部
绝大多数的Activity都是采用的standard的启动模式，
开启新的Activity，这个Activity就会被加载到任务栈的栈顶。
返回退出Activity，就从任务栈里面移除。如果任务栈清空，应用程序退出。

应用场景：绝大多数默认的情况

singletop 单一顶部模式

顶部只有一个实例存在，如果这个Activity的实例已经被加入到任务栈的栈顶，这个Activity就不会被重新创建,而是复用已经存在这个Activity

应用场景：一个界面避免重复的开启很多次

案例:
浏览器的保存书签功能
登录界面

singleTask 单一任务栈模式

如果任务栈里面已经存在要激活的Activity,就不会重新创建.
直接复用这个已经存在的Activity,并且把这个Activity上面其他的所有的Activity清空掉.

应用场景：
如果一个Activity的内存开销或者是cpu开销非常大。
一般我们设置这个Activity为singletask的启动模式。保证任务栈里面只有一个实例存在。

使用案例:
1.适合做应用程序的主界面(MainActivity)，app内部直接startActivity跳转方便
2.浏览器的页面(BrowserActivity),内存开销非常大，因为webkit 浏览器的内核(管理浏览器核心操作) 用于:解析html标签，解析css标签，JavaScript解析引擎

singleInstance 单一实例 模式

在整个手机操作系统里面就只有一个实例存在,
并且这个实例是在他自己单独的任务栈里面,不会用默认的任务栈.

被设置为singleInstance模式的Activity会自己开辟一个任务栈,独立使用
保证页面在系统只有1个
如果希望整个android系统中某个Activity永远是唯一的,可以设置singleInstance模式

目的：避免内存浪费

应用场景：
1.有道词典的快速取词
2.电话拨号器（呼叫InCallScreen）（通话界面）
