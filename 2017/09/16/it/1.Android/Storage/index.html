Android —— 数据存储机制总结

发表于 2017-09-16 | 分类于 IT - Android |


很多时候我们的软件需要对处理后的数据进行存储或再次访问，Android为数据存储提供了如下几种数据持久化方式：

1.存文件

2.SharedPerference(参数)

3.SQLite数据库
保存结构化数据
特殊的文件,带索引

4.用第三方的面向对象数据库直接保存Java对象
当我们有大量的相似结构的数据存放,实现增删改查非常方便,推荐使用数据库.

5.Content Provider内容提供者

6.网络（通过网络把数据保存到服务器上）
安卓是基于网络，对各种协议封装都是完善的，可以利用HTTP协议把数据提交到服务器
底层的socket也是支持的，也可以利用tcp和udp把数据提交到服务器上
如：socket的tcp udp , http的httpurlconnection

7.XML
半结构化存储

8.JSON
保存数组和对象

这篇文章主要讲前3种

文件存储

使用openFileOutput和openFileInput方法，或FileInputStream和FileOutputStream)

手机存储目录的划分：手机内部存储(手机内部), 手机外部存储(SD卡)

适用场景：用于存一般的数据

关于Android文件的存储目录的注意事项 :
android下应用程序的路径和javase不同, 应用程序的数据要保存自己的文件夹里面
文件只能存储在自己应用程序私有的文件夹下,别的应用程序无法访问

手机存储目录的划分

手机内部存储

API：Environment.getDataDirectory()

存储目录：file目录 和 cache目录

优点：容量有限,较小,价格比较贵,存储访问速度快，类似电脑的固态硬盘

为什么内部存储空间不需要声明权限?
创建的文件在手机内部存储空间,默认的权限全部都是私有的.
只有自己的应用程序可以访问,
别的应用程序不可以访问.
如果这个文件不想让别人看到,就存到内部存储空间里

书写方式：
1.可以直接写死路径操作文件,但是给写死了,以后换包名时,所有这样写的路径都要跟着改!
如：/data/data/com.star.login/uinfo.txt

2.为了解决这个为难题提供了如下方法:
context.getFilesDir() –> /data/data/[包名]/files –>这里面存储的内容不会被系统回收
context.getCacheDir() –> /data/data/[包名]/cache –>这里面存储的内容,在某些情况下系统可能会自动回收,虽然会自动回收,但是android推荐,用完最好自己尽快删除

手机外部存储

API：Environment.getExternalStorageDirectory()

存储目录：存到手机的SD上的data/data目录的包名里面

优点：容量大,可以更换,价格便宜,32G 不到200块

使用注意：记得添加权限（读写2个权限）
写入的权限：android.permission.WRITE_EXTERNAL_STORAGE
读取的权限：android.permission.READ_EXTERNAL_STORAGE

写法：
1./mnt/sdcard/…
这种写法不行,因为很多山寨手机sd卡目录可能不叫sdcard
此时可以通过Enviroment工具类获取sd卡的目录位置

这个API指向当前系统sd卡根目录

注意：
1.写sd是需要权限的
读取sd卡默认不需要权限~!
有些如果手机开启了sd卡的读权限控制,则需要配置读取的权限才能使用
要加就一起加上吧，防止某些手机开启了读权限而不能使用的问题!

2.有的手机可能没有装sd卡,此时需要判断是否挂载SD卡

Environment.MEDIA_MOUNTED // 挂载状态
Environment.MEDIA_UNMOUNTED // 卸载状态

Android中Context中的API

对于文件的操作最好使用Android里面Context封装的API，而不要直接使用File对象：

1.openFileOutput(name, mode);
打开文件输出流（参数分别是文件名和访问权限）
写法：openFileInput(“star.txt”);
地址：/data/data/包名/files/star.txt

2.openFileInput()
打开文件输入流
写法：openFileOutput(“star.txt”, Context.MODE_PRIVATE);
地址：/data/data/包名/files/star.txt

3.getFilesDir()
拿到当前用户的文件目录(绝对路径)获取应用程序files目录
地址：/data/data/包名/files

4.getCacheDir()
获取缓存目录
地址：/data/data/包名/cache
获取缓存数据文件夹的路径,很简单但是知道的人不多,这个路径通常在SD卡上(这里的SD卡指的是广义上的SD卡,包括外部存储和内部存储)Adnroid/data/您的应用程序包名/cache/ 下面.测试的时候,可以去这里面看是否缓存成功.缓存在这里的好处是:不用自己再去手动创建文件夹,不用担心用户把自己创建的文件夹删掉,在应用程序卸载的时候,这里会被清空,使用第三方的清理工具的时候,这里也会被清空.

关于files和cache：
这是谷歌给我们提供了两种应用程序的目录:
files目录存放重要的应用程序数据.手机不会自动清理files目录的文件 (用于放配置信息)
cache目录是存放临时的不重要的数据.这个目录有特定,当手机内存空间不足的时候会自动清理cache目录的文件 (用于放临时文件)

如图所示
清除数据是files(注意,有提醒 是否清除账户和数据库等等, 很重要 所以要谨慎清除)
清除缓存是cache(没有提醒, 直接清除,cache文件夹消失)

关于文件访问权限：

默认情况下所有的文件创建出来都是私有的。
只有自己的应用程序可以访问里面的数据，别的应用程序是不可以访问数据的。

特殊情况利用api可以修改文件的权限：
openFileOutput(“文件名”, “文件的访问模式”); // 私有 只读 只写 可读可写
权限的访问模式 : ↓

Linux下权限说明：共10位，drwxrwxrwx （）
第0位d:表示文件类型，d代表文件夹，l代表快捷方式
第1-3位rwx:表示使用者的权限，r可读，w可写，x可执行.
第4-6位rwx:表示组内的权限，r可读，w可写,x可执行.
第7-9位rwx:表示其他人的权限，r可读,w可写,x可执行.

文件的权限10数位可通俗的理解就4种：文件类型 自己 组内其他 路人

CMD下操作权限 (底层是通过Linux操作系统的文件模式来实现的)
chmod : change mode 修改模式 (只能执行在模拟器上,真机上无法执行(因为没刷root权限))
chmod 777

SharedPreferences

简介

所属包：android.content
含义：共享参数
存放位置：/data/data//shared_prefs/目录下的

概述：
轻量级的存储类，特别适合用于保存软件配置参数。
存配置文件,是手机内部存储的XML文件
底层就是XML, 可以快速存储键值对

用途：
非常广泛，只要想保持数据的配置信息，害怕有乱码，非法字符，可以用这个接口把它们都存起来共享参数
还有用在存设置信息, 配置信息, 密码
如果有布尔类型的值，int类型的小数据就存到这里
而且不会有异常产生

缺点：
如果数据特多就不适合用它, 适合用数据库

注意：
1.putXXX（把数据存起来，写入键值信息,但是并没有真的写入）,
2.commit（提交时才真的写入提交数据，才被保存( 类似关闭流,事务 )，一定不要忘记）
3.在多进程之间不要用SharedPreferences共享数据，虽然可以（MODE_MULTI_PROCESS），但极不稳定

关于Editor

Editor 编辑器接口

关于commit方法：
注意SharedPreferences.Editor.commit这个方法是同步的，一直到把数据同步到Flash上面之后才会返回，由IO操作的不可控，尽量使用apply方法代替。
apply只在API Level>=9才会支持，需要做兼容。
不过，最新的 support v4 包已经为我们做好了处理，使用 SharedPreferencesCompat.EditorCompat.getInstance().apply(editor) 即可。

关于apply方法：
所有的commit操作使用了SharedPreferencesCompat.apply进行了替代，目的是尽可能的使用apply代替commit
先说下为什么，因为commit方法是同步的，并且我们很多时候的commit操作都是UI线程中，毕竟是IO操作，尽可能异步；
所以我们使用apply进行替代，apply异步的进行写入；
但是apply相当于commit来说是new API呢，apply的效率高一些，如果没有必要确认是否提交成功建议使用apply。
为了更好的兼容，所以做了适配
apply注意事项：
1.apply没有返回值,而commit返回boolean表明修改是否提交成功
2.apply是将修改数据原子提交到内存, 而后异步真正提交到硬件磁盘, 而commit是同步的提交到硬件磁盘
3.apply方法不会提示任何失败的提示apply的效率高一些，如果没有必要确认是否提交成功建议使用apply。

SQLite数据库

简介

概述：
SQLite，是一款轻量型的数据库，是遵守ACID(原子性、一致性、隔离性、持久性)的关联式数据库管理系统，多用于嵌入式开发中。

Android系统自带的轻量级、运算速度极快的嵌入式关系型数据库。
它不仅支持标准的SQL语法，还可以通过Android封装好的API进行操作，让存储和读取数据变得非常方便。

与Web常用的数据库Mysql、Oracle db、sql server不同，Sqlite是一个内嵌式的数据库，数据库服务器就在你的程序中，无需网络配置和管理，数据库服务器端和客户端运行在同一进程内，减少了网络访问的消耗，简化了数据库管理。

SQLite的数据类型：Typelessness(弱类型), 可以保存任何类型的数据到你所想要保存的任何表的任何列中.
但它又支持常见的类型比如: NULL, VARCHAR, TEXT, INTEGER, BLOB, CLOB…等.
唯一的例外：integer primary key 此字段只能存储64位整数

在Android系统，还提供了一个SQLiteOpenHelper抽象类，该类用于对数据库版本进行管理.该类常用的两个方法:
onCreate 数据库创建
onUpgrade 数据库更新

特点：
安卓内部的数据库，移动平台轻量级嵌入式的数据库
lite: 轻量级
嵌入式 : 不是台式机,就是小一点的平台
完全开源
支持中文, 因为安卓是UTF-8
存储大量的结构相同的数据,工作效率很高

缺点：
Sqlite在并发、数据库大小、网络方面存在局限性，并且为表级锁，所以也没必要多线程操作。

数据库文件路径：
/data/data//databases/目录下

涉及到的API：
SQLiteOpenHelper
SQLiteDatabase
Cursor

常用sql语句

SQLite可以解析大部分标准SQL语句，只需掌握简单的SQL语句即可

创建表：
create table person(
_id integer primary key autoincrement,
name varchar(20),
age integer);
// person是表名

修改表结构：
alter table person add balance integer;

增加一条记录：
规则：insert into 表名(字段列表) values(值列表)
insert into person(name) values(‘tom‘, 25);

删除一条记录：
规则：delete from 表名 where 条件子句
delete from person where name = ‘tom’;

修改（更新）一条记录：
规则：update 表名 set 字段名=值 where 条件子句
update person set name = ‘lisi’ where name = ‘tom’;

查询数据：
语法：select from 表名 where 条件子句 group by 分组字句 having … order by 排序子句
select from person;
select from person order by id desc
select name from person group by name having count()>1

分页SQL与mysql类似，下面SQL语句获取5条记录，跳过前面3条记录：
select from Account limit 5 offset 3 或者 select from Account limit 3,5

获取添加记录后自增长的ID值：
SELECT last_insert_rowid()

获取表中的总记录数：
select count(1) from talbename
查询速度快高效

数据库的增删改查的2种方式

1.自己写sql实现增删改查的操作

优点：sql可以非常的灵活,可以执行任意形式的语句
缺点：需要自己写sql稍微麻烦些,容易出错. 而且没有返回新增行号和影响到的行数

2.利用谷歌api实现增删改查（android提供的进行增删改查的方法）

内部工作的原理：组拼sql语句
优点：省去拼接sql的操作,减少错误 2.可以返回新增的行号 或 影响到的行数
缺点：不能执行过于复杂的sql,比如多表查询..子查询…

总结：
推荐大家使用谷歌的api实现增删改查,有返回值处理方便,避免了手写sql语句出错的问题 (如语句单词写错, 多写个空格等等)
